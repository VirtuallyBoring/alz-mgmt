# Azure Landing Zones Management

This repository contains the Terraform configuration for deploying and managing an enterprise-scale [Azure Landing Zones](https://aka.ms/alz) platform. It was generated by the [ALZ Terraform Accelerator](https://aka.ms/alz/acc) and is designed to be operated entirely through GitHub Actions — not run manually via the Terraform CLI.

## What Gets Deployed

- Management group hierarchy with custom architecture (`alz_custom`)
- Azure Policy definitions and assignments across the hierarchy
- Role definitions and subscription placement
- Management resources in the management subscription:
  - Log Analytics workspace
  - User-assigned managed identity (for Azure Monitor Agent)
  - Data collection rules (change tracking, VM insights, Defender for SQL)
- Microsoft Defender for Cloud configuration (MDFC)

Connectivity (hub-and-spoke VNets or Virtual WAN) is currently disabled (`connectivity_type = "none"`) and can be enabled when needed.

## How It Works with alz-mgmt-templates

This repo's CI/CD workflows are thin wrappers that delegate to reusable workflow templates in [`virtuallyboring/alz-mgmt-templates`](https://github.com/virtuallyboring/alz-mgmt-templates):

```
ci.yaml  ──calls──►  alz-mgmt-templates/.github/workflows/ci-template.yaml@main
cd.yaml  ──calls──►  alz-mgmt-templates/.github/workflows/cd-template.yaml@main
```

This separation keeps workflow logic in one place. Updates to CI/CD behavior are made in `alz-mgmt-templates`, not here.

### CI Pipeline (on every PR to `main`)

1. `validate` — `terraform fmt -check`, `terraform init -backend=false`, `terraform validate`
2. `plan` — `terraform init` with real backend, `terraform plan`, posts result as a PR comment

### CD Pipeline (on every merge to `main`)

1. `plan` — `terraform init`, `terraform plan -out=tfplan`, bundles output as a GitHub artifact
2. `apply` — downloads artifact, `terraform apply -auto-approve tfplan`

Both jobs use **OIDC (keyless) authentication** to Azure — no stored credentials.

## Prerequisites

### Azure

- Four subscriptions: management, connectivity, identity, security (IDs are set in `terraform.tfvars.json`)
- A storage account for Terraform remote state (created by the accelerator bootstrap)
- A service principal or managed identity with sufficient permissions across all subscriptions (created by the accelerator bootstrap)
- OIDC federated credentials on the service principal for GitHub Actions (see below)

### GitHub

Two **Environments** in this repo's settings (`Settings → Environments`):

| Environment | Used by |
|---|---|
| `alz-mgmt-plan` | CI validate/plan job, CD plan job |
| `alz-mgmt-apply` | CD apply job — consider adding a required reviewer |

Six **Variables** in this repo's settings (`Settings → Secrets and variables → Actions → Variables`):

| Variable | Description |
|---|---|
| `AZURE_CLIENT_ID` | Service principal client ID |
| `AZURE_SUBSCRIPTION_ID` | Management subscription ID |
| `AZURE_TENANT_ID` | Azure AD tenant ID |
| `BACKEND_AZURE_RESOURCE_GROUP_NAME` | Resource group containing the tfstate storage account |
| `BACKEND_AZURE_STORAGE_ACCOUNT_NAME` | Storage account name for Terraform state |
| `BACKEND_AZURE_STORAGE_ACCOUNT_CONTAINER_NAME` | Container name (e.g. `tfstate`) |

### OIDC Federated Credentials

In the Azure Portal, navigate to the service principal's **Federated credentials** and add two entries:

| Field | Plan credential | Apply credential |
|---|---|---|
| Issuer | `https://token.actions.githubusercontent.com` | `https://token.actions.githubusercontent.com` |
| Subject | `repo:virtuallyboring/alz-mgmt:environment:alz-mgmt-plan` | `repo:virtuallyboring/alz-mgmt:environment:alz-mgmt-apply` |

## Making Configuration Changes

All day-to-day configuration lives in two files:

### `platform-landing-zone.auto.tfvars`

The primary configuration file. Edit this for:

- **Regions** — `starter_locations` (primary is `centralus`, secondary is `westus2`)
- **Resource naming** — `custom_replacements.names`
- **Policy assignments** — `management_group_settings.policy_assignments_to_modify`
- **Management resources** — `management_resource_settings`
- **Connectivity** — `connectivity_type` (`"none"` | `"hub_and_spoke_vnet"` | `"virtual_wan"`)
- **Tags** — `tags`

### `terraform.tfvars.json`

Edit this for:

- **Subscription IDs** — `subscription_ids` map (management, connectivity, identity, security)
- **Root management group** — `root_parent_management_group_id`

### `lib/` folder

Edit the YAML files here to customize the management group hierarchy or policy assignments beyond what the tfvars support. See the [ALZ Library documentation](https://azure.github.io/Azure-Landing-Zones-Library/) for schema reference.

## Deployment Workflow

1. Create a branch and make your changes
2. Open a pull request to `main` — CI will run and post a Terraform plan as a PR comment
3. Review the plan output carefully before merging
4. Merge the PR — the CD pipeline runs automatically and applies the changes

To trigger a deployment manually: **Actions → 02 Azure Landing Zones Continuous Delivery → Run workflow**

To destroy infrastructure: use the manual trigger and select `destroy` as the action.

## Terraform State

State is stored in the Azure Storage account configured via the `BACKEND_*` GitHub variables. The state key is `terraform.tfstate`. Only one plan or apply job can run at a time — the workflows use a `mgmt-tfstate` concurrency lock to enforce this.
